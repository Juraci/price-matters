# Codebase Concerns

**Analysis Date:** 2026-05-10

## Tech Debt

**Vite scaffold leftover (`counter` store):**

- Issue: `src/stores/counter.ts` is a Pinia counter generated by the Vue/Vite template. It is not imported anywhere, has no tests, but is still persisted (`{ persist: true }`), so it allocates a `counter` key in `localStorage` on every install.
- Files: `src/stores/counter.ts`
- Why: Never removed after the initial scaffold.
- Impact: Dead code; minor `localStorage` pollution; misleads new contributors searching the stores folder.
- Fix approach: Delete the file. Confirm no imports with `grep -r "useCounterStore" src e2e`. No migration needed for users (the orphan key in `localStorage` is harmless).

**README.md is the Vite template:**

- Issue: `README.md` is the unchanged Vite scaffold (sections about Volar, hot reload, generic test commands). The actual project description (CSV import, brapi quotes, Pinia persistence) is documented only in `CLAUDE.md`.
- Files: `README.md`
- Why: Never customised after `npm create vue@latest`.
- Impact: Anyone landing on the GitHub page (or returning to the repo months later) gets no signal about what the app does. Onboarding-friction for collaborators.
- Fix approach: Rewrite the README with a one-paragraph product summary, install/run commands, and pointers to `CLAUDE.md` and `.specs/codebase/` for deeper docs.

**Persisted Pinia state has no schema versioning or migration:**

- Issue: `pinia-plugin-persistedstate` is configured with defaults — every store's full state is rehydrated from `localStorage` on load with no version tag and no migrator.
- Files: `src/main.ts:14-18`, every `defineStore(..., { persist: true })`
- Why: Convenience choice during initial development.
- Impact: When `Ticker`, `TickerSnapshot`, `Empresa`, `ImportBatch`, or `configStore` shapes change in a future release, returning users will hydrate stale shapes and either crash on access (e.g. a renamed field, a removed enum variant) or silently drift. Already a real risk: `cotacaoAtual` was historically inside `TickerSnapshot` and was moved to `Ticker` — anyone who persisted state under the old shape would have lost their live quote on upgrade.
- Fix approach: Add a `version` field to each persisted state and supply a `serializer` (or a custom `afterHydrate` hook) that runs migrations. At minimum, on hydrate-failure, fall back to `reset()` rather than letting an error propagate.

**`brapiApiKey` survives `reset()` but isn't documented in-store:**

- Issue: The fact that `configStore` is intentionally not wiped by the "Limpar dados" button is documented in `CLAUDE.md` but not in the store itself. A future contributor who adds `configStore.reset()` to `CsvImport.handleReset` would silently break the "API key persists across data wipes" guarantee.
- Files: `src/components/CsvImport.vue:41`, `src/stores/configStore.ts`
- Why: Convention encoded in the call sites, not the store contract.
- Impact: Silent regression risk during refactors.
- Fix approach: Either (a) intentionally do not expose any reset on `configStore` (already the case — keep that and add a code comment), or (b) split the brapi key out of `configStore` into a dedicated `secretsStore` that only the user can clear.

## Known Bugs

**`slugify` regex character class looks wrong:**

- Symptoms: For company names containing accented Latin characters (e.g. `"Klabin S/A"`, `"Itaú"`, `"São Paulo"`), `slugify` is meant to strip the combining diacritical marks left by `.normalize('NFD')`. The current regex literal `/[̀-ͯ]/g` (visible in `src/utils/stockUtils.ts:7`) appears to use raw combining marks as range endpoints rather than the canonical Unicode escape `/[̀-ͯ]/g`. Whether this works correctly depends on how the file bytes were saved; a copy-paste through a tool that normalises whitespace can silently break the range.
- Trigger: Re-save the file via a non-UTF-8-safe tool, or re-type the regex by hand.
- Files: `src/utils/stockUtils.ts:7`
- Workaround: None needed today (output looks correct on inspection of test fixtures), but the regex is fragile.
- Root cause: Inline raw combining-mark range instead of `̀-ͯ`.
- Fix approach: Replace with `replace(/[̀-ͯ]/g, '')` and add a unit test in `src/utils/__tests__/stockUtils.spec.ts` covering `Itaú → itau`, `São Paulo → sao-paulo`, and `Petrobras → petrobras`.

**`parseBRNumber` swallows malformed values as `0`:**

- Symptoms: Any unparseable cell (e.g. `"-"`, `"N/A"`, an empty string after a stray comma) produces `0` rather than a parse error. A snapshot with `lucroLiquidoEstimado: 0` and `quantidadeTotalAcoes: 0` will silently be accepted, producing `lucroPorAcaoEstimado = 0`, `dividendYieldBruto = 0`, and a misleading "no margem" reading.
- Trigger: Upload a CSV where any numeric cell is missing or non-numeric.
- Files: `src/utils/csvParser.ts:1-7`
- Workaround: User notices unexpected zeros in the table.
- Root cause: `isNaN(num) ? 0 : num` swallows the parse failure.
- Fix approach: Either (a) propagate `NaN` and let `computeDerived` guard, or (b) collect parse warnings into the `ImportBatch` stats and surface them in the UI alongside `newTickers`/`updatedTickers`/`removedTickers`.

## Security Considerations

**brapi API key stored in plaintext in `localStorage`:**

- Risk: `configStore.brapiApiKey` is persisted to `localStorage` via `pinia-plugin-persistedstate`. Any script with same-origin access (e.g. a malicious browser extension, or an XSS vulnerability if one is ever introduced) can read it.
- Files: `src/stores/configStore.ts:117`, `src/main.ts:14-18`
- Current mitigation: This is a static SPA with no user-authored input rendered as HTML, so XSS surface is low. The brapi token is rate-limited and free-tier only — exposure cost is bounded.
- Recommendations: (a) Document this trade-off explicitly in `SettingsPopover.vue`'s help text. (b) If a paid brapi tier is ever used, consider rotating tokens regularly. (c) Avoid logging `configStore.brapiApiKey` to the console.

**No Content Security Policy:**

- Risk: `index.html` ships no `<meta http-equiv="Content-Security-Policy" ...>` and there's no server to set CSP headers. If an XSS hole is ever introduced (e.g. via `v-html` on user input), there is no mitigation.
- Files: `index.html`
- Current mitigation: The app currently has zero `v-html` usage and renders only structured data through `{{ }}` interpolation — Vue auto-escapes, so the practical risk is near zero today.
- Recommendations: When/if a static host is chosen (Netlify, Cloudflare Pages, etc.), configure a strict CSP at the host level. Until then, this is a low-priority hardening.

## Fragile Areas

**`csvParser.ts` is coupled to fixed column indices:**

- Files: `src/utils/csvParser.ts:50-83`
- Why fragile: Columns are addressed by index (`cols[0]`, `cols[2]`, `cols[15]`, etc.) instead of by header name. A single inserted column upstream — even a cosmetic one — silently shifts every subsequent value.
- Common failures: An upstream spreadsheet edit (added column, reordered columns) produces wrong values without throwing. Symptom: every row's `precoTeto` becomes the previous column's value, making `margemSeguranca` wildly off.
- Safe modification: When the source CSV format changes, edit both the index map AND the test fixtures (`e2e/fixtures/stocks-sample-v[12].csv`, `SAMPLE_CSV` in `csvParser.spec.ts`) in the same commit. Add a unit test that asserts the header row's expected column count and key headers (`Empresa`, `Código`, `Cotação atual`, `Preço Teto`).
- Test coverage: Decent (`csvParser.spec.ts` covers the happy path and a malformed-header throw), but **no test fails when a column is inserted upstream**. Consider switching to header-name addressing.

**Cross-store coupling in `importStore.importCsv`:**

- Files: `src/stores/importStore.ts:14-61`
- Why fragile: `importStore` reaches into both `empresaStore` and `tickerStore` to orchestrate an atomic-feeling operation that is not actually atomic — if `markRemovedIfNotIn` throws after `upsertTicker` mutations have run, the stores are left in a partially updated state.
- Common failures: None observed in production (no exception path inside those calls today), but a future change to `markRemovedIfNotIn` could leave half-imported data.
- Safe modification: Treat all three store mutations inside `importCsv` as a single transaction conceptually. If you add a step that can throw, also add a rollback path or a "staging" pattern (build the new state in temporary objects, swap into the store at the end).
- Test coverage: `importStore.spec.ts` covers the happy path. There are no tests for partial-failure recovery.

**`upsertTicker` returns the literal string union but `importStore` only checks two branches:**

- Files: `src/stores/importStore.ts:38-47`, `src/stores/tickerStore.ts:12-42`
- Why fragile: The function returns `'new' | 'updated' | 'unchanged'`. The caller does `if (result === 'new') ... else if (result === 'updated') ... else stats.unchangedTickers++;` — the `else` correctly catches `'unchanged'`, but it would also silently swallow any future return value (e.g. `'reactivated'`).
- Safe modification: If you add a new return variant, also add an explicit branch in `importStore.importCsv` and a corresponding stat. TypeScript will not catch this regression because `else` is exhaustive at the type level.
- Test coverage: `importStore.spec.ts` covers `new`/`updated`/`unchanged`/`removed` counters but does not assert exhaustiveness.

## Test Coverage Gaps

**`src/services/brapiClient.ts` has no unit tests:**

- What's not tested: `fetchOne`'s handling of non-OK responses, malformed JSON bodies, missing `regularMarketPrice`, and the `Promise.allSettled` reduction in `fetchQuotes`.
- Risk: A regression in the silent-failure semantics (the explicit "skip the ticker, keep previous price" contract) would be invisible until users notice stale prices. The CORS/SDK constraint documented in `CLAUDE.md` ("do not reintroduce the brapi npm SDK", "token in query string not header") has no automated guard.
- Priority: Medium.
- Difficulty to test: Low — `fetch` can be mocked at the global scope (`vi.stubGlobal('fetch', ...)`) and the function is pure-ish.

**`src/views/HomeView.vue` and `src/router/index.ts` have no dedicated tests:**

- What's not tested: That `/` mounts `HomeView`, that `HomeView` mounts both `CsvImport` and `StockDataTable`.
- Risk: Low (router has one route, HomeView is trivial), but if a second route is ever added without test scaffolding, navigation regressions become silent.
- Priority: Low.
- Difficulty to test: Low. E2E coverage is implicit in every `e2e/*.spec.ts` since they all start at `/`.

**No test coverage tooling:**

- What's not tested: there is no measurement of which lines are tested. `npm run test:unit` runs Vitest but not in coverage mode.
- Risk: Cannot tell whether new code is tested without manual inspection.
- Priority: Low for the current size of the codebase.
- Difficulty to test: Trivial — Vitest has `--coverage` built in via `@vitest/coverage-v8`. Add a `test:unit:coverage` script and optionally run it in CI as informational (do not gate the build until a baseline exists).

**No CI step for `format:check` failure beyond exit code:**

- What's not tested: When Prettier check fails, CI prints a diff but does not auto-suggest the fix on the PR.
- Risk: Low; only impacts contributor experience.
- Priority: Low.
- Difficulty to test: Add a comment-bot step to suggest `npm run format` on PR if `format:check` fails.

## Dependencies at Risk

**Major version dependencies (Vite 8, Vitest 4, ESLint 10, TypeScript 6, vue-tsc 3, jsdom 29):**

- Risk: This project tracks the bleeding edge of the Vue/Vite ecosystem. Several listed versions are higher than the latest stable as of typical knowledge cutoffs (e.g. `typescript: "~6.0.0"`, `vite: "^8.0.8"`, `eslint: "^10.2.1"`, `vitest: "^4.1.4"`). Plugin compatibility windows on bleeding-edge majors are short — an `npm install` after a long pause may pull a transitive update that breaks the lint or build pipeline.
- Impact: A future `npm ci` could fail or produce different lint/type-check output than the locked snapshot.
- Migration plan: Treat `package-lock.json` as load-bearing. Run `npm ci` (not `npm install`) in CI (already the case in `.github/workflows/ci.yml:11`). When upgrading any of these, do it as an isolated commit and run the full Build gate (`lint && type-check && format:check && test:unit && build && test:e2e`) before merging.

**`microdiff` produces opaque diff records:**

- Risk: `getDiff` casts results aggressively (`d.path[0] as keyof TickerSnapshot`, `d.oldValue as number`, `d.value as number` in `TickerHistoryDialog.vue:135-141`). If a future snapshot field is non-numeric, those casts will produce nonsense without a runtime error.
- Files: `src/utils/stockUtils.ts:31-34`, `src/components/TickerHistoryDialog.vue:128-152`
- Migration plan: Add a per-field type guard in the dialog's `diffRows` computation, or restrict `VOLATILE_FIELDS` to numeric fields by typing.

## Missing Critical Features

**No way to delete a single ticker without wiping all data:**

- Problem: The only "delete" affordance is `CsvImport.handleReset`, which clears all imports/empresas/tickers at once. A user who wants to remove a single legacy ticker (e.g. one delisted years ago) has no path.
- Current workaround: Re-import a CSV without that ticker — `markRemovedIfNotIn` will flag it as `'removed'`, but it stays in the table (with `opacity: 0.5`). It is never garbage-collected.
- Blocks: Long-term store growth; users with many imports accumulate `'removed'` rows indefinitely.
- Implementation complexity: Small — add a per-row delete affordance in `StockDataTable.vue` that calls a new `tickerStore.deleteTicker(codigo)` action and a corresponding cleanup in `empresaStore` for orphaned company entries.

**No CSV format documentation:**

- Problem: The exact source format the parser expects is documented only via the test fixture and column-index comments. A new user without access to the source spreadsheet has no way to produce a compatible file.
- Current workaround: Read the test fixtures and `csvParser.ts` to reverse-engineer the format.
- Blocks: Onboarding additional users; ability to ingest from sources other than the original spreadsheet.
- Implementation complexity: Small — add a `docs/csv-format.md` with a column-by-column description, or expose a sample-CSV download link from the import button.

**No export path:**

- Problem: All data lives in `localStorage`. There is no "export to JSON / CSV / clipboard" affordance, so the data can be lost if `localStorage` is cleared or the user switches browsers.
- Current workaround: None.
- Blocks: Backups, multi-device usage, sharing.
- Implementation complexity: Small for a JSON dump (`JSON.stringify(stores)` to a download blob); medium for a clean re-importable CSV that round-trips through `parseCsv`.
